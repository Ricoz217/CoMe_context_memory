# CoMe ContextMemory 项目介绍

本文档用于说明 `CoMe_ContextMemory` 的核心设计、能力边界与使用建议。

---

## 1. 基本原理

`ContextMemory` 把一个上下文容器视作一个“记忆桶（bucket）”，并在本地维护同步的存储、缓存与检索索引。

每次 LLM 交互可以抽象为三部分：
1. `system_prompt`：链路级稳定提示
2. `memory`：记忆事件流（Append-Only）
3. `command`：本轮任务指令与 payload

核心思想是 `Event Append-Only`：
1. 不在原地修改历史事件
2. 新状态通过追加事件形成
3. 通过重建/压缩流程得到可读的“最新视图”

---

## 2. 与 RAG 的对比

### 2.1 优点
1. 同一桶内可直接携带完整上下文，不强依赖切片召回。即: RAG无全局观，`ContextMemory` 有且很准。
2. 不需要独立向量数据库即可运行检索链路，因为 `Context` 本身就是一种向量数据库。
3. 桶结构与记忆关系显式可控，便于人工干预与调试。

### 2.2 缺点
1. 受 LLM 上下文窗口限制，深层子桶场景存在路由难题。
2. 树状递归查询在层级较深时会增加延迟与调用成本。
3. 与“固定一次召回+一次回答”的典型 RAG 相比，时延波动更大。

---

## 3. 入库与结构维护

### 3.1 记忆入库
1. 支持 `add_memory`、`add_memory_from_file`、`add_memory_from_dir`。
2. `add_memory_from_file` 支持文本和图片，图片可通过 `image_extract_hint` 指导解析。
3. 暂不支持 `pdf/docx`。

### 3.2 自动维护
1. 上下文压力超过阈值时，可触发 `compress/split_bucket`。
2. 可手动调用 `optimize` 做重排优化。
3. 批量接口默认带自动切片/自动压缩、分桶/自动优化等逻辑。

### 3.3 层级限制
1. 桶深度受 `max_bucket_depth` 控制。
2. 自动创建子桶时会进行深度校验，避免非法嵌套。

---

## 4. Query 主链路

一次 `query` 由“本地检索 + LLM + 本地融合”组成，即: 两次本地计算 + 一次LLM请求。

### 4.1 流程概览
1. 本地路由增强：按预算执行子树扫描（BFS），计算本地召回辅助分。
2. LLM 查询：结合 key hints 与桶上下文生成 `answer + matches`。
3. 本地重排：融合 LLM 分与本地分，得到最终排序。
4. 若命中桶节点，递归查询子桶并将结果回传到顶层结果中。
5. 一句话: BFS -> LLM -> Rerank，同时自动递归上述链路。

### 4.2 本地检索技术
1. 主干：`BM25`
2. 增强：`char 3-gram`（Dice 相似度）
3. 用途：子树路由增强 + 候选精排纠偏

### 4.3 Query 模式
公开模式仅支持：
1. `auto`
2. `semantic`
3. `hybrid`

`auto` 规则：
1. 字面特征强（代码符号/路径等）优先路由到 `hybrid`
2. 普通自然语言路由到 `semantic`

## 5. 成本与延迟

### 5.1 成本来源
主要成本来自 LLM 请求，项目做了 `cache_hit` 优化。  
本地流程通过缓存和批处理优化降低重复请求成本。  
为了成本和延迟控制，强烈推荐唯一指定使用 `DeepSeek V4-flash`，并且走官方API。  
能获得极低的成本和较好的延迟表现，且拥有1M上下文和优异的召回率。  
若要用其他模型或本地LLM也支持，核心LLM请求链路走的是两种标准LLM API，在配置文件注明API格式即可。即:  

- Openai Completion Api
- Anthropic Api   

不支持 `Openai Response Api`

### 5.2 成本详解

成本由不同LLM链路、不同桶的冷热启动影响，冷热启动区别较大，原因如下: 
1. 不同LLM链路的 `system_prompt` 不同，无法命中 `cache_hit`。
2. 每个桶每个链路第一次请求/API供应商缓存过期，属于冷启动，无法命中 `cache_hit`。
3. 在不触发压缩、分桶（即桶一个桶），重复执行某个LLM链路，属于热启动，同时供应商的 `cache` 也会保留更长时间。
4. 热启动成本极低，即: `system_prompt` 和已缓存的老 `memory` 会命中 `cache_hit`，DSV4的缓存命中价格仅为 **0.02元/M token**。
5. 热启动实际消耗的成本，为新增部分的 `memory` 和本次操作的 `command`，这些实际上都很少，且下次请求新增的 `memory` 也会纳入缓存。  

> 一句话: 在桶重建(压缩/分桶)前，以合适的频率 `query` 基本不花钱

### 5.3 延迟来源
1. 入库链路：批量切分、清洗、ingest 调用
2. 查询链路：递归深度、子树扫描预算、是否命中缓存

### 5.4 延迟详解

- 入库较慢，特别是文件入库、目录入库，这些都属于批量入库。原因: 
  1. `ContextMemory` 是单写者模式，且同一个桶不支持并发写，将在下方 `IO限制` 中细讲。
  2. 目录入库是自动遍历调用文件入库。文件入库默认会自动切片，然后创建批量入库任务。
  3. 批量入库任务实际上也是调用单个记忆入库接口(`add_memory`)，只是清洗逻辑不同，且在入库后会自动 `optimize` (注意：会创建一个新桶)。
  4. 每个记忆入库都需要LLM进行 `ingest`，主要是生成关系信息 `relations`。
  5. 批量入库任务有并发优化，但延迟依旧不低。
  6. 入库主要延迟就在LLM请求。
- Query 延迟主要在子树深度，同时与是否冷启动有关。原因: 
  1. `query` 会自动递归 `query` 子桶(子树)。
  2. 热启动(供应商有缓存)LLM请求会显著加快。
  3. `query` 延迟是所有LLM请求的总耗时，多个子桶递归 `query` 会并发请求。
  4. `query` 本身支持并发，桶只有写锁，无读锁。

### 5.5 实践建议
1. 按功能或模块组织子桶，减少无效递归。
2. 优先在目标子桶查询，避免盲扫全树。

## 6. IO 与并发约束

### 6.1 运行模型
1. 设计目标是单写者安全模型。
2. 同一 `BASE_DIR` **不能** 多进程/多实例同时写入。
3. 项目本身没做Redis、分布式锁、多进程锁。用了全异步设计，原子写，但多写者不安全，同时注意多线程的 `event_loop` 处理。
4. 项目本身不支持但 **没限制** 多进程、多实例对同一个记忆库写，请 **千万不要** 这么做。 
5. 提供了 `get_context_memory_engine` 方法获取引擎单例，但也可以通过直接创建 `ContextMemoryEngineV3` 对象获取多个不同实例。

### 6.2 锁语义

写入
: `add/update/move/split/compress/create_bucket/optimize` 等会改变桶内容、创建新桶的都属于写入操作。  

读取
: `qeury/list/get` 等不改变桶内容的，属于读取操作。

1. 同桶写操作串行（不保证严格 FIFO）
2. 跨桶写操作可并发（锁定涉及桶）
3. 读操作可并发（`query/list/get`）
4. 批量入库任务: `add_memory_from_file/add_memory_from_dir` 等，会将一个文件的记忆chunk视为一个批次，等待全部处理完成后一次性入库。  
做了批量 `cache_hit` 优化，并发入库优化，同时会自动触发压缩/分桶/优化。

### 6.3 风险提示
若多个写者共享同一 `BASE_DIR`，可能导致：
1. 索引不一致
2. 缓存映射错乱
3. 记忆库损坏

---

## 7. 接口形态

项目提供三种使用方式：
1. Python API（推荐）: 作为python库导入使用，全对象设计，可直接对象操作。
2. CLI: 单独进程，通过CLI转换至python接口。
3. JSON-RPC 2.0（本地服务）: 单独进程，也是转换接口。为其他语言且不想用CLI提供一种使用方式。

RPC 默认无鉴权，若对外暴露请自行在网关层补充鉴权与限流。

---

## 8. 当前边界与 TODO

当前边界：
1. 无 GUI
2. 暂无分布式锁/多写者协同
3. 记忆库仅为本地文件系统，暂未提供数据库接口
4. 记忆库暂无自动化清理逻辑，仅提供了手动清理接口 `gc_storage`，且本地文件清理周期长，有空间膨胀风险。
5. 虽然记忆和桶有哈希版本管理，但记忆库本身没有
6. 暂无时间管理，仅有部分 LLM 链路会传入记忆创建时间

后续方向（示意）：
1. Web 可视化界面
2. LLM Wiki 形态升级
3. 增量存储与历史管理优化
4. 更完整的自动化清理与运维能力
5. 更完善的记忆管理: 显式权重、过期、锁定
